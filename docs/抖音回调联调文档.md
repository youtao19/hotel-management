# 抖音回调联调文档

## 1. 文档范围
本文档用于说明抖音 OTA 回调接口的本地联调与幂等处理规则，覆盖以下接口：
- `POST /api/douyin/callback/spi`
- `POST /api/douyin/callback/presale`
- `POST /api/douyin/callback/bookable`
- `POST /api/douyin/callback/hotel-info`
- `POST /api/douyin/callback/spi/mock`

## 2. 回调地址
- 正式验签回调：`POST /api/douyin/callback/spi`
- 正式验签预售回调：`POST /api/douyin/callback/presale`
- 正式验签可订检查回调：`POST /api/douyin/callback/bookable`
- 正式验签酒店静态信息处理结果回调：`POST /api/douyin/callback/hotel-info`
- 本地联调免签回调：`POST /api/douyin/callback/spi/mock`
- 本地联调免签可订检查回调：`POST /api/douyin/callback/bookable/mock`
- 本地联调免签酒店静态信息处理结果回调：`POST /api/douyin/callback/hotel-info/mock`
- 抖音联调平台要求回填的 `logID`，取自请求头 `x-bytedance-logid`
- 系统现在会把该值记录到 `douyin_orders.douyin_log_id`

## 3. 幂等规则
- 抖音回调落地表 `douyin_orders` 以 `ota_order_id` 作为幂等唯一键。
- 本地系统订单以 `order_source='douyin' + id_source=ota_order_id` 作为幂等口径。
- 本地订单表 `orders` 按入住天数拆分多行，其中 `stay_date = check_in_date` 的首日行作为主行。
- 当回调重复到达，或首次创建系统订单成功但回写 `douyin_orders.synced` 失败时，后端会：
  - 优先查询本地是否已存在对应系统订单主行；
  - 若已存在，则直接回写 `synced=true` 与 `system_order_id=orders.order_id`；
  - 不会再次创建重复订单。

## 4. 成功响应说明
- `syncResult.action=created`：首次成功创建本地订单并完成回写。
- `syncResult.action=skip`：`douyin_orders` 已标记同步完成，直接跳过。
- `syncResult.action=recovered`：检测到本地订单已存在，执行同步状态补偿回写。

## 4.1 创单失败闭环说明
- 当创单失败但请求里已带 `order_id` 时，系统会尽量把失败信息落到 `douyin_orders`。
- 当前失败排查字段：
  - `booking_stage`
  - `booking_error_code`
  - `booking_error_description`
  - `booking_failure_response`
- 当前失败阶段口径：
  - `payload_invalid`
  - `room_mapping_failed`
  - `inventory_failed`
  - `system_create_failed`
- 这一步不会改变抖音侧响应结构，只补本地失败闭环和排障信息。

## 5. 官方字段与本地映射
- `order_id` -> `douyin_orders.ota_order_id` -> `orders.id_source`
- `source_order_id` -> `douyin_orders.source_order_id`
- `hotel_id` -> `douyin_orders.hotel_id`
- `room_id` -> `douyin_orders.room_id` -> 房型映射查询条件
- `rate_plan_id` -> `douyin_orders.rate_plan_id`
- `biz_type` -> `douyin_orders.biz_type`
- `check_in_date/check_out_date` -> `douyin_orders.check_in_date/check_out_date` -> `orders.check_in_date/check_out_date`
- `number_of_units` -> `douyin_orders.room_count`
- `number_of_guests` -> `douyin_orders.number_of_guests`
- `total_amount` -> `douyin_orders.amount`
- `amount_before_tax` -> `douyin_orders.amount_before_tax`
- `daily_rates` -> `douyin_orders.daily_rates` -> 本地 `roomPrice`
- `occupancies` -> `douyin_orders.occupancies`
- `contact_info.name` -> `douyin_orders.contact_name`
- `contact_info.phone` -> `douyin_orders.contact_mobile`
- `remark_from_douyin` -> `douyin_orders.remark_from_douyin`
- `remark_from_guest` -> `douyin_orders.remark_from_guest`
- `member_info` -> `douyin_orders.member_info`

## 6. 创单响应结构
- 当前系统已按抖音“酒店创建订单”文档返回 `data.error_code`、`data.description`、`data.order_id`、`data.order_out_id`、`data.confirm_info`。
- `confirm_info.confirm_mode` 当前默认返回 `2`，表示异步接单模式。
- 如请求体传入 `confirm_mode=1`，或环境变量 `DOUYIN_CONFIRM_MODE=sync`，则会返回同步接单模式 `1`。
- 重复回调时，如果本地订单已存在，仍返回成功，不返回“重复提交预定请求”错误。

成功示例：

```json
{
  "data": {
    "error_code": 0,
    "description": "success",
    "order_id": "DY_ORDER_001",
    "order_out_id": "O202603240001",
    "confirm_info": {
      "confirm_mode": 2
    }
  }
}
```

失败示例：

```json
{
  "data": {
    "error_code": 1,
    "description": "房型不存在/失效",
    "order_id": "DY_ORDER_001"
  }
}
```

## 7. 加密字段说明
- `contact_info.phone`、`occupancies[].phone`、`occupancies[].license_id` 按官方文档可能为加密字段。
- 当前系统会原样落库到 `douyin_orders` 的相关字段和 `mapped_payload/raw_payload`。
- 由于仓库中的解密实现尚未完成，本阶段不会将加密串直接转换为本地明文手机号。
- 后续接入正式解密能力后，再补齐“明文手机号入本地订单”的逻辑。

## 8. 联调建议
- 首次联调建议优先使用 `/api/douyin/callback/spi/mock`，便于跳过签名校验，专注验证回调业务逻辑。
- 取消订单联调建议使用 `/api/douyin/callback/cancel/mock`，用于验证本地取消、幂等取消和异步审核分支。
- 同一笔抖音订单重复回调时，应观察返回值中的 `syncResult.action` 是否符合预期。
- 若历史测试遗留了“本地订单已创建但 `douyin_orders.synced=false`”的数据，新版本接口会自动补偿恢复。
- 联调验收页面要求填写 `logID` 时，可直接查询：

```sql
SELECT ota_order_id, douyin_log_id, system_order_id, created_at
FROM douyin_orders
ORDER BY created_at DESC
LIMIT 20;
```

## 9. 错误码与状态枚举
- 创单错误码：
  - `1`：房型不存在/失效
  - `4`：入住时期内已满
  - `5`：日期格式错误
  - `6`：姓名/联系电话格式错
  - `13`：其他参数或业务异常
  - `100`：需要重试
- 取消错误码：
  - `9`：订单不存在或状态异常
  - `13`：参数或其他异常
- 响应枚举：
  - `confirm_mode=1`：同步接单
  - `confirm_mode=2`：异步接单
  - `cancel_mode=2`：异步取消
- 取消状态枚举：
  - `cancelled`：本地已取消
  - `already_cancelled`：本地订单已是取消状态
  - `pending_audit`：等待抖音售后审核回传
  - `invalid_status`：本地订单状态不允许直接取消
  - `missing_local_order`：找不到对应本地订单
- 审核推送状态枚举：
  - `pending_push`：已生成审核结论，待推送
  - `sent`：审核结果已成功回传抖音
  - `push_failed`：审核结果回传失败，需后续补偿
## 10. 取消订单 SPI 联调说明
- 正式验签回调：`POST /api/douyin/callback/cancel`
- 本地联调免签回调：`POST /api/douyin/callback/cancel/mock`
- 当前系统统一返回 `cancel_mode=2`，表示异步取消模式。
- 处理规则：
  - `pending/reserved`：本地直接改为 `cancelled`
  - `cancelled`：按幂等成功处理
  - `checked-in/occupied/checked-out`：进入异步审核状态，不直接改本地状态
  - `need_audit=true`：进入异步审核状态
- 当 `need_audit=true` 时，系统会在本地事务提交后自动调用抖音“售后审核结果返回” OpenAPI：
  - `pending/reserved/cancelled`：回传“同意取消”
  - `checked-in/occupied/checked-out`：回传“拒绝取消”
- 审核回传失败不会回滚本地取消 SPI 处理结果，但会记录到 `douyin_orders.cancel_audit_*` 字段，便于后续补偿重试。

成功示例：

```json
{
  "data": {
    "error_code": 0,
    "description": "success",
    "cancel_mode": 2,
    "reason": "用户取消"
  }
}
```

失败示例：

```json
{
  "data": {
    "error_code": 9,
    "description": "订单不存在或状态异常",
    "cancel_mode": 2
  }
}
```

## 11. 联调测试项
- 创单字段缺失时，应返回对应业务错误码，不返回 500。
- 创单重复回调时，应按幂等更新处理。
- 无可用房时，应返回 `error_code=4`。
- 取消回调 `need_audit=true` 时，应自动触发审核结果回传。
- 审核结果回传失败时，本地取消状态不回滚，但 `cancel_audit_status` 应写为 `push_failed`。

## 12. 确认接单/拒单联调说明
- 调用地址：`POST /api/douyin/order/confirm`
- 当前支持两种回传结果：
  - `confirmResult=1`：接单
  - `confirmResult=2`：拒单
- 参数说明：
  - `otaOrderId`：抖音订单号，必填
  - `confirmResult`：确认结果，必填
  - `confirmNumber`：接单确认号；当 `confirmResult=1` 时必填
  - `rejectCode`：拒单原因码；当 `confirmResult=2` 时选填，默认 `1`
  - `rejectReason`：拒单原因；当 `confirmResult=2` 时选填，默认“酒店暂不可接单”
- 本地回写规则：
  - 接单成功：`douyin_orders.confirm_status='confirmed'`
  - 拒单成功：`douyin_orders.confirm_status='rejected'`
  - 抖音接口失败：`douyin_orders.confirm_status='failed'`
- 保护规则：
  - 若订单在创单时已记录 `confirm_mode=1`，则不允许再调用本接口做异步确认

接单示例：

```json
{
  "otaOrderId": "DY_ORDER_001",
  "confirmResult": 1,
  "confirmNumber": "CN_202603240001"
}
```

拒单示例：

```json
{
  "otaOrderId": "DY_ORDER_002",
  "confirmResult": 2,
  "rejectCode": 9,
  "rejectReason": "库存不足"
}
```

## 13. 预售券创单 SPI 联调说明
- 正式验签回调：`POST /api/douyin/callback/presale`
- 本地联调免签回调：`POST /api/douyin/callback/presale/mock`
- 当前系统用于承接 `biz_type=2011` 的住宿预售券创单请求。
- 处理规则：
  - 预售券创单成功时，会同时写入 `douyin_orders` 和 `douyin_presale_orders`
  - 接口成功响应中的 `order_out_id` 来源于 `douyin_presale_orders.order_id`
  - 预售券本地单不会直接拆分成 `orders` 住宿日订单，也不会立即占房
  - 后续预约单可通过 `source_order_id` 关联这笔预售券订单

成功示例：

```json
{
  "data": {
    "error_code": 0,
    "description": "success",
    "order_id": "DY_PRE_001",
    "order_out_id": "O202603240001"
  }
}
```

## 14. 可订检查 SPI 联调说明
- 正式验签回调：`POST /api/douyin/callback/bookable`
- 本地联调免签回调：`POST /api/douyin/callback/bookable/mock`
- 当前系统先支持 `biz_type=2011/2012/2021` 的基础可订检查。
- 当前校验逻辑：
  - `rate_plan_id` 必须能反查到本地抖音物理房型
  - 抖音物理房型必须已映射到本地房型
  - 本地房型必须存在且未关闭
  - 本地可售房量必须大于等于 `number_of_units`
  - 发单价格需与本地房型基础价一致
- 当前错误码口径：
  - `0`：可订成功
  - `1`：房型不存在/失效
  - `4`：入住时期内已满
  - `5`：日期格式错误
  - `8`：价格与酒店实际价格不一致
  - `13`：其他参数或业务异常
  - `18`：日历房态关闭
- 当返回不可订，系统会同步返回 `data.ari.stock_and_amount`，用于抖音侧及时更新价量态。

成功示例：

```json
{
  "data": {
    "error_code": 0,
    "description": "success"
  }
}
```

## 14.1 酒店静态信息处理结果推送Webhook 联调说明
- 正式验签回调：`POST /api/douyin/callback/hotel-info`
- 本地联调免签回调：`POST /api/douyin/callback/hotel-info/mock`
- 当前定位：模式二（自助匹配）下的补充能力，用于接收异步结果并稳定回执 success。
- 模式二主链路接口：
  - `POST /api/douyin/hotel-info/query`
  - `POST /api/douyin/hotel-info/sync`
  - `GET/POST /api/douyin/room-type-mapping`
- 房型映射口径：
  - 一个本地房型只能绑定一个抖音物理房型
  - 一个抖音物理房型只能绑定一个本地房型
  - 保存接口会先返回清晰业务错误，数据库唯一约束只做并发兜底
- 当前处理逻辑：
  - 正式回调使用 Webhook 签名头 `X-Douyin-Signature` 做验签（`sha1(client_secret + rawBody)`）
  - 从请求头提取 `x-bytedance-logid` 与 `Msg-Id`
  - 读取请求体中的酒店标识（`hotel_id` 或 `out_hotel_id`）
  - 当 `event=verify_webhook` 时，按要求回传 `{"challenge":"xxx"}`
  - 对 `Msg-Id` 做内存去重，重复消息直接返回 success
  - 打印结构化日志，便于联调排查
  - 返回统一结构 `data.error_code/data.description`
- 当前校验规则：
  - `hotel_id` 与 `out_hotel_id` 至少传一个
  - 两者都缺失时返回业务错误 `error_code=13`
- 当前阶段说明：
  - 当前先完成 webhook 接收与回执，保证回调联调不阻塞
  - 酒店静态信息状态落库、状态机联动将在后续阶段补充

成功示例：

```json
{
  "data": {
    "error_code": 0,
    "description": "success"
  }
}
```

失败示例：

```json
{
  "data": {
    "error_code": 13,
    "description": "缺少酒店标识"
  }
}
```

## 15. 物理房型创建调试说明
- 调用地址：`POST /api/douyin/physical-room/create`
- 当前用于手动创建抖音物理房型，便于后续继续做商品创建、上下架和价量态映射。
- 请求参数：
  - `localRoomType`：本地房型编码，必填
  - `poiId`：抖音酒店 ID，必填
  - `categoryId`：抖音类目 ID，必填
  - `images`：房型图片列表，必填；当前至少传 1 张图片，字段支持 `imageUrl`
  - `accountId`：抖音商家账号 ID，选填；未传时回退使用环境变量 `DOUYIN_ACCOUNT_ID`
- 当前创建逻辑：
  - 读取本地 `room_types` 获取房型名称、描述、关闭状态
  - 统计本地 `rooms` 中该房型的房间数量，写入 `room_num`
  - 以 `type_code` 作为 `out_room_id`
  - 创建成功后会立即调用“物理房型静态信息查询”回查详情
  - 回查成功后，写入 `douyin_physical_rooms` 与 `douyin_room_type_mapping`
- 当前最小字段口径：
  - `cn_name`：本地 `room_types.type_name`
  - `room_num`：本地该房型房间数量
  - `active`：`room_types.is_closed=false`
  - `max_occupancy`：当前固定 `2`
- 若同一本地房型已存在抖音物理房型映射，接口会直接返回业务错误，不重复创建。

请求示例：

```json
{
  "localRoomType": "asu_xiao_zhu",
  "poiId": "7198765432101234567",
  "categoryId": "8001001",
  "images": [
    {
      "imageUrl": "https://example.com/room-1.jpg"
    }
  ]
}
```

## 15.1 物理房型上下架调试说明
- 调用地址：`POST /api/douyin/physical-room/status`
- 当前用于手动切换抖音物理房型的上架/下架状态。
- 请求参数：
  - `roomId`：抖音物理房型 ID，必填
  - `active`：上下架状态，必填布尔值
  - `accountId`：抖音商家账号 ID，选填；未传时回退使用环境变量 `DOUYIN_ACCOUNT_ID`
- 当前处理逻辑：
  - 先读取本地 `douyin_physical_rooms`
  - 通过 `physical_room/save` 更新 `active`
  - 抖音返回成功后，再把本地 `status` 和 `raw_payload.active` 回写更新
  - 抖音返回失败时，不修改本地状态
- 当前限制：
  - 这是基础版“物理房型上下架”
  - 不包含删除物理房型
  - 不会级联更新商品上下架状态

请求示例：

```json
{
  "roomId": "ROOM_001",
  "active": false
}
```

## 16. 日历房商品创建调试说明
- 调用地址：`POST /api/douyin/rate-plan/create`
- 当前用于手动创建抖音日历房商品，便于继续推进“创建日历房商品”验收项。
- 请求参数：
  - `localRoomType`：本地房型编码，必填
  - `poiId`：抖音酒店 ID，必填
  - `roomId`：抖音物理房型 ID，必填
  - `accountId`：抖音商家账号 ID，选填；未传时回退使用环境变量 `DOUYIN_ACCOUNT_ID`
  - `mode`：商品模式，选填；当前支持 `meal`、`cancel`、`stay`、`booking`
  - `modeConfig`：模式配置，选填；不同模式支持不同字段
- 当前创建逻辑：
  - 通过本地房型名称生成 `rate_plan_name`
  - 通过 `localRoomType + mode` 生成 `out_rate_plan_id`
  - 创建成功后，将返回的 `rate_plan_id` 合并写回 `douyin_physical_rooms.rate_plan_list`
- 当前基础字段口径：
  - `currency='CNY'`
  - `active` 跟随本地 `room_types.is_closed`
  - `confirm_immediately=false`
  - `sales_type=1`
  - `settle_type=1`
- 当前模式配置：
  - `meal`：支持 `modeConfig.mealCount`
  - `cancel`：支持 `modeConfig.freeCancelHoursBeforeCheckIn`
  - `stay`：支持 `modeConfig.minStayNights`、`modeConfig.maxStayNights`
  - `booking`：支持 `modeConfig.advanceBookingDaysMin`、`modeConfig.advanceBookingDaysMax`
- 当前限制：
  - 这是复杂商品规则第一阶段，不代表已完整覆盖全部动态商品字段
  - 当前仍未覆盖会员价、营销价、多段取消政策、多段预约窗口等更复杂场景

请求示例：

```json
{
  "localRoomType": "asu_xiao_zhu",
  "poiId": "7198765432101234567",
  "roomId": "ROOM_001",
  "mode": "meal",
  "modeConfig": {
    "mealCount": 2
  }
}
```

## 16.1 本地套餐同步抖音商品说明
- 调用地址：
  - `GET /api/douyin/rate-plans`
  - `POST /api/douyin/rate-plan/sync`
- `GET /api/douyin/rate-plans` 用于同步页面读取本地套餐列表：
  - 主表为 `rate_plans`
  - 左联 `rooms`、`room_types`、`douyin_room_type_mapping`、`douyin_physical_rooms`、`ota_channel_mappings`
  - 返回 `localDataStatus` 区分 `OK`、`ROOM_MISSING`、`ROOM_TYPE_MISSING`
  - 返回 `localRoomExists`、`localRoomTypeExists`，页面可明确展示“本地数据异常”
  - 返回 `douyinRoomId`、`douyinRatePlanId`、`syncStatus`
  - `syncStatus` 第一版只表示已落库的同步事实：成功为 `1`，未同步为 `null`；同步失败当前不落库
- 当前用于把本地 `rate_plans` 套餐同步为抖音日历房商品。
- 请求参数：
  - `localRatePlanId`：本地套餐 ID，必填，对应 `rate_plans.id`
  - `poiId`：抖音酒店 ID，选填；未传时优先使用已同步物理房型 `raw_payload` 中的酒店 ID，再回退环境变量 `DOUYIN_POI_ID`
  - `accountId`：抖音商家账号 ID，选填；未传时优先使用已同步物理房型账号，再回退环境变量 `DOUYIN_ACCOUNT_ID`
  - `mode`：商品模式，选填；当前支持 `meal`、`cancel`、`stay`、`booking`
  - `modeConfig`：模式配置，选填；字段规则同 `POST /api/douyin/rate-plan/create`
- 当前同步逻辑：
  - 从 `rate_plans -> rooms -> room_types` 读取本地套餐和所属房型
  - 通过 `douyin_room_type_mapping` 找到本地房型绑定的抖音物理房型
  - 通过 `douyin_physical_rooms` 读取抖音物理房型缓存和酒店 ID
  - 使用本地 `rate_plans.id` 作为 `out_rate_plan_id`
  - 调用抖音 `rateplan/save` 创建或更新商品
  - 成功后回写 `douyin_physical_rooms.rate_plan_list`
  - 成功后 UPSERT `ota_channel_mappings`，唯一键为 `local_target_type + local_target_id + channel_code`
- 幂等规则：
  - 首次同步会插入 `ota_channel_mappings`
  - 再次同步同一个本地套餐到抖音时，只更新 `channel_item_id`、`channel_config`、`sync_status`、`updated_at`
  - 抖音返回业务失败时，不写入本地映射表
- 当前限制：
  - 依赖本地房型已经先绑定抖音物理房型
  - 当前本地 `rate_plans` 只有基础字段，餐食、取消、连住、预定限制等复杂规则仍通过 `modeConfig` 传入

请求示例：

```json
{
  "localRatePlanId": 101,
  "poiId": "7198765432101234567",
  "mode": "meal",
  "modeConfig": {
    "mealCount": 2
  }
}
```

成功响应示例：

```json
{
  "success": true,
  "action": "synced",
  "mode": "meal",
  "ratePlan": {
    "rate_plan_id": "RATE_001",
    "out_rate_plan_id": "101"
  },
  "douyinRatePlanId": "RATE_001",
  "savedMapping": {
    "local_target_type": "RATE_PLAN",
    "local_target_id": 101,
    "channel_code": "DOUYIN",
    "channel_item_id": "RATE_001",
    "sync_status": 1
  }
}
```

## 16. 入住/离店状态同步调试说明
- 当前系统已补充一组手动调试接口，用于“入住/离店状态同步能力”基础联调：

  - `POST /api/douyin/order/check-out`
- 请求体统一结构：

```json
{
  "orderId": "O202603240001"
}
```

- 当前同步口径：
  - 入住：推送 `accommodation_status=1`
  - 离店：推送 `accommodation_status=3`
  - 抖音订单号来源：`douyin_orders.ota_order_id`
  - 第三方订单号来源：`douyin_orders.system_order_id`
- 本地回写字段：
  - `douyin_orders.fulfillment_status`
  - `douyin_orders.fulfillment_action`
  - `douyin_orders.fulfillment_response`
  - `douyin_orders.fulfillment_sent_at`
- 当前已支持：
  - 手动按本地 `orderId` 推送入住
  - 手动按本地 `orderId` 推送离店
  - 本地办理入住成功后自动推送
  - 本地办理退房成功后自动推送
- 自动触发规则：
  - 仅抖音来源订单会触发
  - 自动同步失败不会回滚本地入住/退房
  - 失败结果仍会记录到 `douyin_orders.fulfillment_*`
  - noshow 场景 `accommodation_status=2`

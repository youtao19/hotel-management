# 订单唯一约束修改说明

**日期**: 2025-10-13  
**版本**: v1.1  
**状态**: ✅ 已完成

---

## 📋 修改概述

将订单表的唯一约束从**房型**改为**房间号**，以防止同一客人在相同时间段重复预订相同房间。

### 原约束
```sql
UNIQUE (guest_name, check_in_date, check_out_date, room_type)
```

### 新约束
```sql
UNIQUE (guest_name, check_in_date, check_out_date, room_number)
```

---

## 🎯 修改原因

### 1. 业务逻辑更合理
- **旧约束问题**: 允许同一客人在同一时间段预订同一房型的不同房间，但限制了预订不同房型
- **新约束优势**: 直接约束实际的房间，更符合酒店业务实际情况

### 2. 实际场景分析

#### 旧约束场景
```javascript
// ❌ 旧约束允许（不合理）
订单1: { 姓名: "张三", 日期: "2025-10-15~16", 房型: "yi_jiang_nan", 房间号: "205" }
订单2: { 姓名: "张三", 日期: "2025-10-15~16", 房型: "yi_jiang_nan", 房间号: "206" } // 可以创建

// ✅ 旧约束阻止（合理）
订单1: { 姓名: "张三", 日期: "2025-10-15~16", 房型: "yi_jiang_nan", 房间号: "205" }
订单2: { 姓名: "张三", 日期: "2025-10-15~16", 房型: "yi_jiang_nan", 房间号: "205" } // 被阻止
```

#### 新约束场景
```javascript
// ✅ 新约束允许（合理）
订单1: { 姓名: "张三", 日期: "2025-10-15~16", 房型: "yi_jiang_nan", 房间号: "205" }
订单2: { 姓名: "张三", 日期: "2025-10-15~16", 房型: "xing_yun_ge", 房间号: "206" } // 可以创建

// ❌ 新约束阻止（合理）
订单1: { 姓名: "张三", 日期: "2025-10-15~16", 房型: "yi_jiang_nan", 房间号: "205" }
订单2: { 姓名: "张三", 日期: "2025-10-15~16", 房型: "any_type", 房间号: "205" } // 被阻止
```

## 🔧 修改内容

### 1. 数据库表结构修改

#### 文件: `backend/database/postgreDB/tables/order.js`

```javascript
// 旧约束
CONSTRAINT unique_order_constraint UNIQUE (guest_name, check_in_date, check_out_date, room_type)

// 新约束
CONSTRAINT unique_order_constraint UNIQUE (guest_name, check_in_date, check_out_date, room_number)
```

#### 部分唯一索引（重要改进）
```javascript
// 新增：只对活跃订单应用唯一约束
CREATE UNIQUE INDEX uniq_orders_active ON orders (
  guest_name, check_in_date, check_out_date, room_number
) WHERE status NOT IN ('cancelled', 'checked-out')
```

**说明**: 
- 表级约束：对所有订单（包括历史订单）生效
- 部分索引：只对活跃订单生效，允许历史数据重复

### 2. 代码逻辑修改

#### 文件: `backend/modules/orderModule.js`

**修改点 1: checkExistingOrder 函数**
```javascript
// 旧代码
const { guest_name, check_in_date, check_out_date, room_type } = orderData;
WHERE room_type = $4

// 新代码
const { guest_name, check_in_date, check_out_date, room_number } = orderData;
WHERE room_number = $4
```

**修改点 2: 错误消息**
```javascript
// 旧消息
'该客人在相同时间段已有相同类型的房间预订'

// 新消息
`该客人在相同时间段已有相同房间(${room_number})的预订`
```

### 3. 数据库迁移脚本

**文件**: `backend/database/postgreDB/migrations/2025-10-13-change-unique-constraint-to-room-number.sql`

执行步骤：
1. 删除旧的唯一约束
2. 删除旧的部分唯一索引
3. 创建新的唯一约束（基于房间号）
4. 创建新的部分唯一索引（只对活跃订单）

## ⚠️ 影响分析

### 1. 触发唯一键冲突的条件

#### 新约束下会被阻止的情况
当创建订单满足以下**所有条件**时会失败：
- ✅ 客人姓名相同
- ✅ 入住日期相同
- ✅ 退房日期相同
- ✅ **房间号相同**（新）

#### 不会冲突的情况
以下任何**一个**条件不同即可创建：
- 不同客人姓名 ✅
- 不同入住日期 ✅
- 不同退房日期 ✅
- 不同房间号 ✅（新）
- ~~不同房型~~（不再作为约束条件）

### 2. 对现有功能的影响

#### ✅ 正面影响
1. **更灵活的订单创建**
   - 同一客人可以在同一时间段预订不同房间
   - 不再受房型限制

2. **更精确的重复检测**
   - 直接约束房间，避免房间重复分配
   - 减少人为错误

3. **历史数据兼容**
   - 部分唯一索引允许历史订单重复
   - 不影响已退房/已取消的订单归档

#### ⚠️ 需要注意的点
1. **前端验证逻辑**
   - 需要确保前端也基于房间号进行重复检查
   - 错误提示需要更新

2. **测试用例**
   - 需要更新相关测试用例
   - 重点测试同一客人多房间预订场景

## 🧪 测试验证

### 测试场景 1: 相同房间冲突
```sql
-- 现有订单
INSERT INTO orders (order_id, guest_name, check_in_date, check_out_date, room_number, room_type, status, ...)
VALUES ('O001', '张三', '2025-10-15', '2025-10-16', '205', 'yi_jiang_nan', 'checked-in', ...);

-- 尝试创建冲突订单（应该失败）
INSERT INTO orders (order_id, guest_name, check_in_date, check_out_date, room_number, room_type, status, ...)
VALUES ('O002', '张三', '2025-10-15', '2025-10-16', '205', 'xing_yun_ge', 'pending', ...);
-- ❌ 错误: 该客人在相同时间段已有相同房间(205)的预订
```

### 测试场景 2: 不同房间允许
```sql
-- 现有订单
INSERT INTO orders (order_id, guest_name, check_in_date, check_out_date, room_number, room_type, status, ...)
VALUES ('O001', '张三', '2025-10-15', '2025-10-16', '205', 'yi_jiang_nan', 'checked-in', ...);

-- 创建不同房间的订单（应该成功）
INSERT INTO orders (order_id, guest_name, check_in_date, check_out_date, room_number, room_type, status, ...)
VALUES ('O002', '张三', '2025-10-15', '2025-10-16', '206', 'yi_jiang_nan', 'pending', ...);
-- ✅ 成功: 允许同一客人预订不同房间
```

### 测试场景 3: 历史订单允许重复
```sql
-- 已退房订单
INSERT INTO orders (order_id, guest_name, check_in_date, check_out_date, room_number, room_type, status, ...)
VALUES ('O001', '张三', '2025-10-15', '2025-10-16', '205', 'yi_jiang_nan', 'checked-out', ...);

-- 相同信息的新订单（应该成功，因为旧订单已退房）
INSERT INTO orders (order_id, guest_name, check_in_date, check_out_date, room_number, room_type, status, ...)
VALUES ('O002', '张三', '2025-10-15', '2025-10-16', '205', 'yi_jiang_nan', 'pending', ...);
-- ✅ 成功: 部分索引不约束历史订单
```

## 1. 当前目标

### 1.1 总目标
- 通过抖音官方直连应用测试验收。
- 完成直连应用上线前所需的核心接口、回调和业务链路。
- 将当前实现状态、缺失功能和开发顺序文档化，作为后续执行基线。

### 1.2 当前聚焦范围
- 优先完成“酒店日历房直连”必测能力。
- 根据验收要求，补齐“商品、价量态、交易、履约、逆向”关键链路。
- 新预售券解决方案相关能力作为后续扩展范围，视验收要求补齐。

- 预售券创单 SPI。

- 取消审核结果回传。
- 异步接单成功链路。
- 异步拒单链路。
- 基础版价量态主链路。
- 基础版入住/离店状态同步。
- 基础版物理房型创建。
- 基础版日历房商品创建。
- 基础版日历房商品上下架。
- 基础版物理房型上下架。
- 基础版主动拉取价量态。
- 基础版退款结果通知。
- 基础版未支付超时取消。
- 抖音 client_token 获取与缓存。
- 抖音 SPI 验签能力。
- 物理房型查询与同步到本地。

### 2.2 当前可以直接或基本开展的测试
- 日历房用户下单后第三方创单成功。
- 日历房创单字段缺失、日期错误、房型未映射、无可用房等失败场景。
- 提单页可订检查成功。
- 可订检查失败。
- 重复回调幂等处理。
- 异步接单成功。
- 异步接单拒单。
- 基础版酒店日历房房价量态更新。
- 基础版入住/离店状态同步。
- 基础版创建物理房型。
- 基础版创建日历房商品。
- 基础版商品上下架。
- 基础版物理房型上下架。
- 基础版主动拉取价量态。
- 基础版退款结果通知。
- 基础版未支付超时取消。
- 取消订单回调。
- 已取消幂等取消。
- 已入住/已离店进入异步审核分支。
- 预售券创单 SPI。

### 2.3 当前只能部分覆盖的测试
- 异步接单超时场景：可以通过不触发确认接单制造超时，但当前没有完整的验收辅助能力。
- 取消退款链路：当前覆盖的是“取消通知 + 审核回传”，不是完整退款闭环。
- 预售券链路：当前只有预售券创单，没有预约单、支付通知、逆向退款等后续能力。

### 2.4 当前明显缺失的关键能力
- 日历房房价/房态/房量更新。
- 同步接单。
- 入住状态同步。
- 离店状态同步。
- 预售券商品管理能力。
- 预售券审核结果通知。
- 预售券预约单相关能力。

## 3. 验收视角下的能力判断

### 3.1 当前可测项
- 用户下单后第三方创单成功。
- 用户下单后第三方创单失败。
- 提单页可订检查。
- 可订检查失败。
- 重复回调幂等。
- 异步接单成功。
- 取消订单 SPI。
- 取消审核结果回传。
- 预售券创单 SPI。
- 创建物理房型。
- 创建日历房商品。
- 商品上下架。
- 物理房型上下架。
- 主动拉取价量态。
- 退款结果通知。
- 不支付超时取消订单。

### 3.2 当前部分可测项
- 用户下单支付后超时未接单。
- 异步取消审核分支。
- 创单异常场景中的一部分价量态异常模拟。

### 3.3 当前无法通过的验收项
- 物理房型上下架。
- 创建日历房商品的餐食信息模式。
- 创建日历房商品的取消政策模式。
- 创建日历房商品的连住模式。
- 创建日历房商品的预定限制模式。
- 价量态异常导致的价格不一致。
- 价量态异常导致的库存不一致。
- 价量态异常导致的房态不一致。
- 商品上下架。
- 酒店日历房房价量态更新。
- 商家主动推送确认入住。
- 商家主动推送确认离店。
- 日历房退款完整 case。
- 用户协商退款取消成功。
- 第三方创单失败后的完整异常链路补偿。
- 不支付超时取消订单。

### 3.4 无法通过的主要原因
- 当前代码重点在“交易回调承接”，不是“全量直连解决方案”。
- 商品侧能力尚未接入，因此大量“创建商品/上下架”验收项无法开始。
- 价量态能力已接入基础版，但“异常场景”和“主动拉取 SPI”还未完成，因此相关验收项仍无法稳定通过。
- 履约状态已接入基础版手动同步，但自动触发和更细场景还未完成，因此相关验收仍需继续补齐。
- 退款逆向链路不完整，因此“退款 case”无法完整闭环。

## 5. 必补功能清单

### 5.1 P0：最影响直连验收通过率
- 可订检查 SPI：已实现待验收，当前为基础版本。
- 酒店日历房房价/房态/房量更新：已实现基础版，仍缺主动拉取 SPI 与异常联调辅助能力。
- 支持同步接单与异步接单的验收要求判断和配置。
- 入住状态同步：已实现基础版手动推送，仍缺自动触发。
- 离店状态同步：已实现基础版手动推送，仍缺自动触发。

### 5.2 P1：商品与房型侧基础能力
- 创建物理房型：已实现基础版，仍需正式验收。
- 物理房型上下架：已实现基础版，仍需继续补删除能力。
- 创建日历房商品：已实现第一阶段复杂规则，仍需继续补更完整商品字段。
- 商品上下架：已实现基础版，仍需继续补删除与批量能力。
- 主动拉取价量态：已实现基础版，仍缺更复杂的营销价和会员价口径。
- 退款结果通知：已实现基础版，仍缺退款到账后的账务联动。
- 未支付超时取消订单：已实现基础版，当前仍为手动调试入口。
- 日历房商品的餐食信息、取消政策、连住模式、预定限制模式。

### 5.3 P1：逆向与异常链路
- 第三方创单失败后的异常闭环。
- 不支付超时取消订单。
- 日历房退款 case。
- 协商退款取消成功。
- 退款结果通知。

### 5.4 P2：新预售券解决方案扩展
- 创建/更新预定商品。
- 创建/更新预售券。
- 预售券审核结果通知。
- 创建预约订单。
- 酒店预约修改订单。
- 支付结果通知。
- 预售券取消退款通知。
- 预售券逆向审核回传。

## 6. 推荐开发顺序

### 6.1 第一阶段
- 先把当前已实现的交易回调链路跑通，确保创单、取消、幂等、异步接单成功都可复现。
- 同时梳理抖音验收后台中的必测项和当前截图中的“是”项。

### 6.2 第二阶段
- 先补可订检查。
- 再补价量态主链路。
- 再补入住/离店状态同步。

### 6.3 第三阶段
- 补物理房型上下架。
- 补日历房商品上下架，并补齐复杂商品规则。
- 补日历房退款完整闭环。

### 6.4 第四阶段
- 视需要补新预售券完整解决方案能力。

# 测试环境修复完成报告

## 问题描述

在修改 `app.js` 将路由注册移到 `initializeSession()` 函数后，所有测试都失败，报错：
```
TypeError: global.cleanupTestData is not a function
```

## 问题根源

### 1. Jest 配置文件路径问题

**核心问题**：从 `backend/` 目录运行 `npm test` 时，Jest 无法找到项目根目录的 `jest.config.js`，导致：
- ❌ `setup.js` 没有被加载
- ❌ `global.cleanupTestData` 未定义
- ❌ 所有测试环境初始化失败

**原因分析**：
```bash
# 项目结构
/hotel-management/
  ├── jest.config.js          # Jest 配置文件在这里
  └── backend/
      ├── package.json        # npm test 从这里运行
      └── tests/
          └── setup.js        # setup.js 在这里
```

当从 `backend/` 运行 `npm test` 时，Jest 会从当前目录查找配置文件，找不到就使用默认配置。

### 2. app.js 结构变更导致的连锁问题

修改后的 `app.js` 将路由注册移到了 `initializeSession()` 函数中：

```javascript
// app.js
async function initializeSession() {
    // ... session 初始化
    
    // ✅ 路由现在在这里注册
    const userRoute = require("./routes/userRoute");
    app.use("/api/user", userRoute);
    // ...
}
```

这要求测试环境也必须调用 `initializeSession()`，否则所有 API 路由都不存在（404）。

## 解决方案

### 1. 修复 Jest 配置加载

**修改 `backend/package.json`**：

```json
{
  "scripts": {
    "test": "cross-env NODE_ENV=test jest --config=../jest.config.js",
    "test:watch": "cross-env NODE_ENV=test jest --config=../jest.config.js --watch"
  }
}
```

**关键点**：添加 `--config=../jest.config.js` 明确指定配置文件路径。

### 2. 在测试 setup 中初始化 app

**修改 `backend/tests/setup.js`**：

```javascript
const db = require('../database/postgreDB/pg');
const redisDB = require('../database/redis/redis');
const app = require('../app');  // ✅ 引入 app

// 定义全局清理函数（必须在 beforeAll 之前）
global.cleanupTestData = async () => {
  try {
    await db.query(truncateTablesSQL());
  } catch (error) {
    console.warn('cleanupTestData 警告:', error.message);
  }
};

// 全局测试设置
beforeAll(async () => {
  // ✅ 初始化 app 的 session 和路由（关键！）
  await app.initializeSession();
  
  // 初始化数据库结构
  await db.initializeHotelDB();
  
  // 初始化 Redis 连接
  await redisDB.initialize();
  
  console.log('✅ 测试环境初始化完成');
}, 60000);
```

**关键改动**：
1. 在 `beforeAll` 中添加 `await app.initializeSession()`
2. 将 `global.cleanupTestData` 定义移到 `beforeAll` 之前（确保立即可用）
3. 增加超时时间到 60 秒

### 3. 正确的测试清理

**修改 `test/auth.test.js`**（示例）：

```javascript
afterAll(async () => {
    // 清理测试数据
    if (testUser) {
        await db.query(deleteUserQuery);
    }

    // 关闭 Redis 连接
    if (redisClient) {
        await redisClient.disconnect();  // ✅ 使用 disconnect()
    }

    // 关闭数据库连接池
    await db.closePool();  // ✅ 使用 closePool()
}, 30000);
```

## 关键知识点

### 1. Jest 配置文件查找规则

Jest 按以下顺序查找配置：
1. `--config` 参数指定的文件
2. 当前目录的 `jest.config.js`
3. `package.json` 中的 `jest` 字段
4. 默认配置

### 2. setupFilesAfterEnv 执行时机

```javascript
// jest.config.js
setupFilesAfterEnv: ['<rootDir>/backend/tests/setup.js']
```

- 在测试框架安装完成后执行
- 在所有测试文件运行前执行
- 可以访问 Jest 全局对象（`describe`, `beforeAll`, `global` 等）
- 适合全局初始化和 teardown

### 3. 测试环境隔离

| 环境 | 入口 | 初始化位置 | 影响 |
|------|-----|-----------|------|
| 开发 | `server.js` | `bootup()` | ❌ 不使用 setup.js |
| 生产 | `server.js` | `bootup()` | ❌ 不使用 setup.js |
| 测试 | Jest | `setup.js` | ✅ 使用 setup.js |

## 最佳实践

### 1. 测试文件组织

```
backend/
  ├── tests/
  │   ├── setup.js           # 全局测试设置
  │   ├── test-helpers.js    # 测试辅助函数
  │   ├── __mocks__/         # Mock 文件
  │   ├── unit/              # 单元测试
  │   ├── integration/       # 集成测试
  │   └── api/               # API 测试
```

### 2. 测试初始化模式

```javascript
// setup.js
beforeAll(async () => {
  // 1. 初始化应用（路由、中间件）
  await app.initializeSession();
  
  // 2. 初始化数据库
  await db.initializeHotelDB();
  
  // 3. 初始化其他服务
  await redisDB.initialize();
});

afterAll(async () => {
  // 1. 清理数据
  await db.query('TRUNCATE ...');
  
  // 2. 关闭连接（顺序重要！）
  await redisDB.close();
  await db.closePool();
});
```

### 3. 避免测试挂起

```javascript
// ✅ 正确关闭连接
afterAll(async () => {
  await redisClient.disconnect();  // 使用 disconnect()
  await db.closePool();            // 关闭连接池
});

// ❌ 错误示例
afterAll(async () => {
  await redisClient.quit();  // 可能不完全关闭
  await db.pool.end();       // pool 可能为 null
});
```

## 注意事项

1. **Jest 配置路径**：使用相对路径 `../jest.config.js`
2. **串行执行**：`maxWorkers: 1` 避免数据库冲突
3. **超时时间**：初始化需要足够时间（60s）
4. **连接清理**：必须正确关闭所有连接
5. **环境隔离**：测试环境不影响开发环境

## 后续优化建议

1. **分离测试类型**：
   - 单元测试：不依赖数据库，可并发执行
   - 集成测试：依赖数据库，串行执行

2. **数据库隔离**：
   - 考虑每个测试使用事务回滚
   - 或使用独立的测试数据库

3. **Mock 策略**：
   - 外部 API 调用使用 mock
   - 减少对真实服务的依赖

4. **CI/CD 集成**：
   - 在 CI 环境中自动运行测试
   - 生成测试报告和覆盖率

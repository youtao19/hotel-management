# Playwright E2E 新手流程（从 0 到跑通 1 条用例）

> 目标：让你用 Playwright 在本项目里**快速写出可维护**的 E2E，并能稳定在本地/CI 跑通。

---

## 0. 先明确：E2E 在测什么

- E2E 关注：用户从页面操作 → 前后端真实交互 → 最终 UI 结果（或接口结果）
- 不要在 E2E 里测太细的业务逻辑细节（那是单元/集成测试的职责）
- E2E 断言优先级：**页面关键文案/金额/状态** > DOM 结构 > CSS 样式

---

## 1. 环境准备（一次性）

1. 确认项目已安装依赖：`npm i`
2. 确认 Playwright 已安装并初始化（如未初始化）：
   - `npx playwright install`
3. 确认你能把项目跑起来（前端/后端/数据库）
   - 通常用 `docker compose` 或项目现有脚本（以项目 README/DOCKER 文档为准）

---

## 2. 了解 Playwright 项目结构（你只需要记住 3 件事）

1. 测试入口：一般是 `playwright.config.*`
2. 测试文件：一般在 `e2e/` 或 `tests/`（以本项目实际为准）
3. 跑测试：`npx playwright test`

> 建议：先跑一次 `npx playwright test --list` 看项目有哪些用例。

---

## 3. 写 E2E 前：先把“数据”固定住（非常关键）

E2E 最常见不稳定来源是数据不确定（环境里订单/账单不同、日期不一致、并发污染）。

建议流程：

1. 选择一个**测试数据库**（不要用开发/生产库）
2. 用固定脚本把数据导入到测试库
   - 本项目要求：先导入 `/sql/orders.csv`
3. 导入后做 1 次快速校验（可选但强烈建议）
   - 校验行数/日期范围是否覆盖 `2025-11-02 ~ 2025-11-07`
   - 校验关键字段（金额、房号、日期）是否存在

> 小技巧：让“导入脚本”成为你 E2E 的 `beforeAll` 或者 CI 的初始化步骤，避免手工导入。

---

## 4. 第 1 条用例怎么写（通用模板）

1. 进入页面：`page.goto(...)`
2. 等页面稳定（用可见元素作为“页面就绪”信号）
3. 做操作：点击/输入/选择日期/提交
4. 做断言：检查关键文本/金额是否出现且正确
5. 截图/trace（失败时帮助定位）

断言建议：

- 选择器优先用 `getByRole()`、`getByLabel()`、`getByText()`（可读性强、稳定）
- 避免依赖 `nth-child`、复杂 CSS 路径（非常脆）

---

## 5. 新手最推荐的写法：先用 codegen 录一遍

如果你对选择器不熟：

- `npx playwright codegen <你的URL>`

录制一次后：

1. 把生成代码粘到测试里
2. 手动把选择器改成更语义化（role/label/text）
3. 把“等待”改成等待关键元素出现（不要到处 `waitForTimeout`）

---

## 6. 登录处理（避免每个用例都重复登录）

常用做法（推荐）：

1. 先手动/自动登录一次
2. 保存 `storageState`（cookie/localStorage）
3. 后续用例复用该状态直接进入系统

关键词命令：

- `npx playwright test --project=...`
- `--headed`（有界面）
- `--debug`（逐步调试）
- `--ui`（可视化运行）

---

## 7. 针对本项目的两类 E2E（你可以按这个拆用例）

### 7.1 收入统计（按日）

前置：导入 `/sql/orders.csv`

断言样例（期望值）：

- `2025-11-02`：`3115.14`
- `2025-11-03`：`3233.97`
- `2025-11-04`：`3462.66`
- `2025-11-05`：`3675.12`
- `2025-11-06`：`2891.76`
- `2025-11-07`：`4845.85`

建议断言策略：

- 先筛选日期（或切换到对应日期/报表视图）
- 再断言“今日收入/当日收入”的金额文本是否等于期望值
- 金额断言注意保留两位小数、千分位、单位（元）等展示差异

### 7.2 “详细收入数据”表格

断言点：

1. 默认展示：**全部账单数据**（不做筛选时行数/分页存在即可）
2. 日期默认：**今日**
3. 可筛选：按**日期 + 房号**筛选后，表格只展示符合条件的账单

建议做法：

- 用“今日”作为默认断言：检查日期控件默认值
- 再切换到一个固定日期（例如 2025-11-02）+ 具体房号，断言表格结果只包含该房号/日期的记录

---

## 8. 调试与稳定性（新手必看）

1. 不要用 `waitForTimeout` 当主等待手段，优先等待元素可见/接口完成
2. 用 trace 定位失败：
   - 运行时加 `--trace on` 或在配置里开启
3. 断言先“存在”再“内容正确”，排查更快

---

## 9. 运行命令速查

- 列出用例：`npx playwright test --list`
- 跑全部：`npx playwright test`
- 跑单文件：`npx playwright test path/to/test.spec.ts`
- 跑单用例：`npx playwright test -g "用例名片段"`
- 带界面：`npx playwright test --headed`
- 调试：`npx playwright test --debug`
- UI 模式：`npx playwright test --ui`

---

## 10. 写完用例后的检查清单

- [ ] 用例不依赖本机时间/随机数据（或已固定）
- [ ] 选择器语义化（role/label/text）
- [ ] 断言稳定（不依赖 DOM 结构细节）
- [ ] 失败能定位（有 trace/screenshot）
- [ ] 数据初始化可重复（脚本化导入 `/sql/orders.csv`）
